# 인증 세션 처리 구조 설계

## 1. 목적

이 문서는 `apps/web`의 인증 처리 구조를 정리하고, `proxy`와 페이지별 `layout`의 책임을 어떻게 분리할지 정의한다.

현재 구조에서는 `proxy`가 `access_token` 존재 여부만 확인하고, 실제 사용자 검증은 각 페이지 `layout`에서 수행하고 있다. 이때 `access_token`이 만료되고 `refresh_token`만 남아 있는 경우, 서버 컴포넌트 단계에서 세션을 복구하지 못해 정상적인 접근 제어가 어려워지는 문제가 있다.

이 문서의 목표는 다음과 같다.

- 세션 복구와 권한 검증의 책임을 분리한다.
- `access_token` 만료 상황에서도 일관된 동작을 보장한다.
- 페이지별 권한 검증 코드를 유지하되, 세션 처리 공통 로직은 중앙화한다.

## 2. 현재 구조

### 2.1 `proxy`의 역할

현재 `proxy`는 다음을 담당한다.

- `/home`, `/login`, `/tournament/join` 진입 시 `access_token`이 없으면 게스트 로그인 수행
- `/archive`, `/mypage/edit`, `/mypage/withdraw` 진입 시 `access_token`이 없으면 로그인 페이지로 리다이렉트
- 멤버 전용 페이지에서 `x-redirect-path` 헤더를 주입해 `layout`이 로그인 리다이렉트 경로를 알 수 있게 함

즉, 현재 `proxy`는 "토큰 존재 여부" 중심으로만 동작한다.

### 2.2 `layout`의 역할

현재 일부 페이지 `layout`은 다음을 담당한다.

- `getMe()` 호출을 통한 현재 사용자 조회
- `identityType`이 `MEMBER`인지 확인
- 401 발생 시 로그인 페이지로 리다이렉트
- 403/404 등 도메인 권한 예외 처리

즉, 현재 `layout`은 "실제 사용자 권한 검증"을 담당한다.

### 2.3 문제점

현재 구조의 핵심 문제는 세션 복구 시점이 비어 있다는 점이다.

예를 들어 멤버 전용 페이지 진입 흐름은 다음과 같다.

1. 브라우저 요청이 `proxy`를 통과한다.
2. `proxy`는 `access_token` 쿠키가 존재하므로 요청을 통과시킨다.
3. 실제로는 해당 `access_token`이 이미 만료된 상태일 수 있다.
4. 페이지 `layout`에서 `getMe()`를 호출한다.
5. 서버 컴포넌트에서 사용하는 `serverApi`는 refresh 재시도를 하지 않는다.
6. `getMe()`가 401로 실패하고, 사용자는 로그인 페이지로 이동된다.

즉, `refresh_token`으로 세션을 복구할 수 있는 상황인데도, 복구 전에 권한 검증이 실행되어 사용자가 불필요하게 로그인 페이지로 이동하는 문제가 발생한다.

## 3. 원인 분석

### 3.1 클라이언트와 서버의 인증 처리 차이

클라이언트의 `clientApi`는 401 응답을 받으면 refresh 요청을 수행하고, 성공 시 원래 요청을 재시도한다.

반면 서버 컴포넌트에서 사용하는 `serverApi`는 단순히 현재 쿠키를 백엔드로 전달만 한다. 따라서 서버 렌더링 단계에서 401이 발생해도 refresh를 통한 세션 복구가 일어나지 않는다.

### 3.2 왜 `layout`에서 세션 복구를 맡기기 어려운가

서버 컴포넌트에서 백엔드 refresh API를 호출하더라도, 그 결과로 내려온 `Set-Cookie`를 최종 브라우저 응답에 안정적으로 반영하기 어렵다.

반면 `proxy`는 `NextResponse`를 직접 생성하므로 다음이 가능하다.

- refresh 응답의 `Set-Cookie`를 그대로 브라우저 응답에 반영
- 이후 `layout`과 서버 API 호출이 최신 세션 기준으로 동작하도록 보장

따라서 세션 복구 책임은 `layout`보다 `proxy`에 두는 것이 적절하다.

## 4. 권장 구조

### 4.1 책임 분리 원칙

권장 구조는 다음과 같다.

- `proxy`
  - 세션 존재 여부 확인
  - 필요한 경우 refresh 또는 guest bootstrap 수행
  - 요청이 이후 단계에서 사용할 수 있는 "유효한 세션"을 확보
- 각 페이지 `layout`
  - 현재 유저가 해당 페이지 권한을 갖는지 검증
  - `MEMBER` 여부, 리소스 접근 가능 여부, 403/404 분기 처리

한 문장으로 정리하면 다음과 같다.

> `proxy`는 세션을 보장하고, `layout`은 권한을 검증한다.

### 4.2 라우트 정책 분류

라우트는 아래 세 가지 정책으로 분류하는 것을 권장한다.

#### 1) `anonymous-allowed`

인증 없이 접근 가능한 라우트다.

- 예: `/`

처리 방식:

- `proxy`에서 아무 작업 없이 통과
- `layout`에서 인증 검증 불필요

#### 2) `guest-or-member`

게스트 또는 멤버 세션만 있으면 접근 가능한 라우트다.

- 예: `/home`, `/login`, `/tournament/join`

처리 방식:

- 유효한 `access_token`이 있으면 통과
- `access_token`이 만료되었고 `refresh_token`이 있으면 refresh 수행
- 둘 다 없으면 guest 로그인 수행

#### 3) `member-required`

멤버만 접근 가능한 라우트다.

- 예: `/archive`, `/mypage/edit`, `/mypage/withdraw`

처리 방식:

- 유효한 세션이 있으면 통과
- `access_token` 만료 + `refresh_token` 존재 시 refresh 수행
- refresh 실패 또는 `refresh_token` 부재 시 로그인 페이지로 리다이렉트
- 이후 `layout`에서 `getMe()` 또는 전용 검증 유틸로 `MEMBER` 여부 확인

## 5. 권장 요청 흐름

### 5.1 게스트/멤버 공통 페이지 진입

1. 요청이 `proxy`에 도달한다.
2. `proxy`가 세션 상태를 확인한다.
3. 유효한 세션이 있으면 그대로 통과한다.
4. `access_token`이 만료되었지만 `refresh_token`이 있으면 refresh를 수행한다.
5. 세션이 전혀 없으면 guest 로그인으로 세션을 발급한다.
6. 페이지는 이미 유효한 세션을 가진 상태로 렌더링된다.

### 5.2 멤버 전용 페이지 진입

1. 요청이 `proxy`에 도달한다.
2. `proxy`가 세션 상태를 확인한다.
3. 필요 시 refresh를 수행한다.
4. 세션 확보에 실패하면 로그인 페이지로 리다이렉트한다.
5. 세션 확보에 성공하면 `x-redirect-path` 같은 보조 헤더를 주입한 뒤 요청을 통과시킨다.
6. 각 페이지 `layout`이 `getMe()` 또는 공통 권한 검증 함수를 호출한다.
7. `identityType !== 'MEMBER'`이면 로그인 또는 적절한 예외 화면으로 분기한다.

## 6. 페이지 `layout`에서 계속 검증해야 하는 이유

세션 확보가 `proxy`에서 끝난다고 해서 페이지 `layout` 검증이 불필요해지는 것은 아니다.

그 이유는 다음과 같다.

- 멤버 전용 페이지는 단순 로그인 여부가 아니라 실제 `MEMBER` 권한 확인이 필요하다.
- 토너먼트 상세/아이템 페이지는 로그인 여부와 별개로 특정 리소스 접근 권한 확인이 필요하다.
- 403, 404, 처리 중 상태 같은 도메인 규칙은 페이지 레벨에서 분기하는 것이 자연스럽다.

따라서 `layout`은 제거 대상이 아니라, 인증보다 한 단계 위의 권한 검증 레이어로 유지하는 것이 맞다.

## 7. 구현 권장 사항

### 7.1 `proxy`가 담당해야 하는 공통 유틸

`proxy`에서는 다음과 같은 공통 유틸 수준의 구조를 두는 것을 권장한다.

- 현재 경로의 접근 정책 판별
- access token 존재 여부 확인
- refresh token 존재 여부 확인
- access token 만료 여부 또는 만료 임박 여부 확인
- refresh 요청 수행
- guest 로그인 수행
- `Set-Cookie` 헤더를 `NextResponse`에 반영

핵심은 "토큰이 있느냐"가 아니라 "이 요청을 처리할 수 있는 유효 세션을 만들 수 있느냐"를 기준으로 판단하는 것이다.

### 7.2 `layout`가 담당해야 하는 공통 유틸

멤버 전용 페이지는 각 레이아웃에서 아래와 같은 공통 유틸을 사용하는 구조를 권장한다.

- `getCurrentUser()`
- `requireMemberUser()`
- `requireAuthorizedTournamentAccess()`

이렇게 하면 각 페이지는 자신의 도메인 규칙만 표현하면 되고, 로그인 리다이렉트와 사용자 타입 검증 로직은 공통화할 수 있다.

### 7.3 `serverApi`의 역할

`serverApi`는 서버에서 현재 요청의 쿠키와 헤더를 백엔드로 전달하는 역할만 유지하는 것이 안전하다.

`serverApi` 자체에 refresh interceptor를 넣는 방식은 다음 이유로 권장하지 않는다.

- 서버 내부에서는 refresh 성공 후 브라우저 쿠키를 안정적으로 갱신하기 어렵다.
- 일시적으로 서버 요청만 성공하고, 브라우저는 여전히 만료된 쿠키를 가진 상태가 될 수 있다.
- 결과적으로 다음 요청에서 같은 문제가 반복될 수 있다.

즉, refresh는 가능하면 브라우저 응답을 직접 제어할 수 있는 `proxy`에서 수행하는 것이 적절하다.

## 8. 기대 효과

이 구조를 적용하면 다음 효과를 기대할 수 있다.

- `access_token` 만료 + `refresh_token` 존재 상황에서 로그인 페이지로 불필요하게 튕기지 않는다.
- 세션 복구 로직이 `proxy`에 모여 인증 처리 흐름이 예측 가능해진다.
- 각 페이지 `layout`은 권한과 도메인 검증에 집중할 수 있다.
- 멤버 페이지와 리소스 보호 페이지의 접근 제어가 일관된 기준으로 동작한다.

## 9. 최종 결론

현재 구조에서는 `proxy`가 토큰 존재 여부만 판단하고, `layout`이 권한 검증을 수행하고 있다. 이 자체는 방향이 크게 틀린 것은 아니지만, `access_token` 만료 상황을 복구하는 책임이 빠져 있다.

권장 구조는 다음과 같다.

- `proxy`는 세션 확보를 담당한다.
- `layout`은 페이지별 권한 및 도메인 접근 검증을 담당한다.

즉, 토큰 갱신이나 세션 bootstrap은 `proxy`에서 처리하고, 실제 페이지 권한 검증은 각 페이지 `layout`에서 계속 수행하는 방향이 가장 적절하다.
